# 非エンジニアが最初にやるべき「たった2つのファイル設定」

> 出典：テツメモ｜AI図解×検証｜Newsletter (@tetumemo)  
> 投稿日：2026年2月25日  
> URL：https://x.com/tetumemo/status/（Feb 25投稿）

---

## 概要

Claude Codeは、AIが実際にファイルを作成・移動・削除・コマンド実行できる強力なツール。  
だからこそ、最初に「やっていいこと」と「ダメなこと」を設定しておく必要がある。

設定ファイルは **たった2つ**、どちらも `.claude` フォルダの中に置く。

| ファイル | 役割 | 例え |
|---|---|---|
| `settings.local.json` | AIの行動を仕組みとして制限 | ドアの「鍵」 |
| `CLAUDE.md` | AIにルールと理由を文章で伝える | 壁に貼った「ルール表」 |

---

## なぜルールが必要か

Claude Codeは普通のチャットAIとは違い、パソコンの操作権限をAIに渡している状態。

**実際に起きた事故例：**
> 「rm -rf ~/」を実行してホームディレクトリを全削除された（devcontainer内で助かった）

人間のアシスタントを雇うときと同じで、最初に「これはやっていいけど、これはダメ」と伝えることが重要。

---

## ① settings.local.json — AIに「鍵」をかける

AIの行動を**仕組みとして**コントロールする。AIがルールを忘れても、物理的に実行できない状態にする。

### 設定例（全文）

```json
{
  "permissions": {
    "allow": [
      "Bash(mkdir:*)",
      "Bash(move:*)",
      "Bash(cp:*)",
      "Bash(node --version:*)",
      "Bash(npm --version)",
      "Bash(npm init -y)",
      "Bash(npm start)",
      "WebFetch(domain:www.pexels.com)",
      "WebFetch(domain:pixabay.com)",
      "WebFetch(domain:code.claude.com)",
      "WebFetch(domain:raw.githubusercontent.com)",
      "WebFetch(domain:agentnativedev.medium.com)"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(rm -rf:*)",
      "Bash(del:*)",
      "Bash(rmdir:*)",
      "Bash(Remove-Item:*)"
    ]
  }
}
```

### ✅ allow（許可リスト）— 確認なしでやっていいこと

#### ファイル操作
| コマンド | 説明 |
|---|---|
| `mkdir` | フォルダを新しく作る（新しい箱を用意するだけ） |
| `move` | ファイルを別の場所に移す（引っ越し） |
| `cp` | ファイルをコピーする（バックアップにも使える） |

#### バージョン確認
| コマンド | 説明 |
|---|---|
| `node --version` | Node.jsのバージョンを見るだけ |
| `npm --version` | npmのバージョンを確認するだけ |

#### プロジェクト操作
| コマンド | 説明 |
|---|---|
| `npm init -y` | プロジェクトの初期設定ファイルを作る（ノートの表紙を作る感覚） |
| `npm start` | プロジェクトを起動する（アプリの「再生ボタン」） |

#### Webアクセス（個人的な追加）
| ドメイン | 用途 |
|---|---|
| Pexels、Pixabay | 無料素材サイト |
| code.claude.com | Claudeの公式ドキュメント |
| raw.githubusercontent.com | GitHubの公開ファイル取得 |
| agentnativedev.medium.com | 参考ブログ |

> リストにない操作はAIが毎回「実行していいですか？」と確認してくる。  
> ソフトのインストールはあえて入れない（毎回説明を受けて自分で判断したいから）。

### ⛔ deny（禁止リスト）— 絶対にやってはダメなこと

| コマンド | 対応OS | 説明 |
|---|---|---|
| `rm` | Mac/Linux | ファイルを消す |
| `rm -rf` | Mac/Linux | フォルダごと強制削除（最も危険） |
| `del` | Windows | ファイルを消す |
| `rmdir` | Windows | フォルダを消す |
| `Remove-Item` | Windows | ファイル・フォルダを消す |

全部「消す」操作。Windows・Mac・Linuxの方言を全部ブロック。  
**ここに入れた操作は、AIがどんなに「必要」と判断しても、システムが自動で止める。**

## ② CLAUDE.md — AIに「ルール表」を見せる

`settings.local.json` が「鍵」なら、こちらは「壁に貼ったルール表」。  
AIがこのファイルを読んで、自分がどう振る舞うべきかを理解する。

### 設定例（全文）

```markdown
## ワークフローのオーケストレーション（日本語版・統合）

### 0. 運用ルール（あなたが入力した内容に対する実行指示）
- あなたが入力した内容は、次のやり方で処理する。
- 作業は「バックグラウンドで動く担当（バックグラウンド担当）」が進める想定にする。
- 一定間隔で状況を確認し、やっていることを短く要約して伝える。
- あなたは技術者ではない前提で、簡潔で分かりやすく説明する。
- 仕組みの説明は「探索担当（解説担当）」を使って、バックグラウンド担当が何をしているか／どう動くかを学べる形で説明する。
- バックグラウンド担当がエラーに遭遇したら、必ずあなたに伝える。
  - その際、「どう直せばいいか」を分かる言葉で案内する。
- すべて日本語で回答する。
- 技術用語は使わない（使わざるを得ない場合は、必ず言い換えを先に置く）。

### 1. Plan ノード（デフォルト）
- どんな「手順が多い作業」でも plan モードに入る（目安：3ステップ以上、または大事な判断が必要なもの）
- 何かがおかしくなったら、すぐに STOP して再計画する（押し切らない）
- plan モードは「作る」だけでなく「確かめる手順」にも使う
- 曖昧さを減らすため、最初に詳細な仕様を書く

### 2. サブエージェント戦略（役割分担）
- メインの作業メモを散らかさないため、役割分担を積極的に使う
- 調査・探索・並列作業は、別の担当に任せる
- 複雑な問題は、担当を増やして同時に進める
- 1担当につき1タスク（同時に抱えない）

### 3. 自己改善ループ
- あなたから修正が入ったら必ず：その学びを tasks/lessons.md に記録する
- 同じミスを防ぐための「自分向けルール」を書く
- ミスが減るまで繰り返し改善する
- セッション開始時に、関係する学びを見直す

### 4. 完了前の検証
- 動くことを証明するまで、完了扱いにしない
- 必要に応じて、変更前と変更後の動きの違いを確認する
- 自問する：「経験豊富な人が見てもOKと言えるか？」
- 確認作業・記録確認・正しさの実演を行う

### 5. エレガンス要求（バランス）
- 手順が多い変更では立ち止まり、「もっとスッキリしたやり方はあるか？」と問う
- 修正が無理やり感あるなら：「今の知識が最初からあるなら、自然な解決にする」
- 単純で明らかな修正はスキップ（やりすぎない）
- 提出前に、自分で厳しく見直す

### 6. 自律的なバグ修正
- バグ報告を受けたら：そのまま直しに行く（手取り足取りを求めない）
- 「どこが変／何が起きているか」を示し、その上で解決する
- あなたに余計な切り替え作業を要求しない
- 指示されなくても、失敗している自動チェックを直しに行く

## タスク管理
1. まず計画：チェックできる項目で plan を tasks/todo.md に書く
2. 計画の確認：実装を始める前にチェックインする
3. 進捗の追跡：進めながら項目を完了にしていく
4. 変更の説明：各ステップで高レベルの要約を書く
5. 結果の文書化：tasks/todo.md にレビューセクションを追加する
6. 学びの記録：修正が入った後に tasks/lessons.md を更新する

## コア原則
- シンプル最優先：あらゆる変更を可能な限りシンプルに。最小限の手数で最大効果。
- 怠けない：根本原因を見つける。一時しのぎは禁止。きちんと直す。
- 最小影響：必要な箇所だけを触る。新しい問題を持ち込まない。

## 安全ルール（ファイル・コマンド操作の保護）

### 7. 既存ファイルの上書き禁止（確認必須）
- 既存ファイルを編集・上書きする前に、必ず「〇〇を上書きしますが、よろしいですか？」と確認する
- 確認なしに既存ファイルの内容を変更しない
- 可能であれば変更前のバックアップを作成する（例：filename.bak）

### 8. 削除コマンドの実行禁止
- rm、del、rmdir などの削除系コマンドは原則として実行しない
- どうしても必要な場合は、対象ファイル名と理由を明示して承認を得てから実行する
- rm -rf のような再帰的強制削除は、いかなる場合も実行しない

### 9. パッケージ追加は事前説明と承認が必要
- npm install、pip install、brew install などのパッケージ追加コマンドは、実行前に以下を説明する：
  - 何をインストールするか（パッケージ名）
  - なぜ必要か（目的・用途）
  - 影響範囲（グローバルかローカルか）
- 説明後、承認を得てから実行する

### 10. 不明なコマンドは実行前に日本語で説明
- 私がエンジニアではないことを常に意識する
- 実行しようとするコマンドが技術的・専門的な場合は、実行前に日本語で以下を説明する：
  - このコマンドは何をするか（平易な言葉で）
  - 実行するとどうなるか（結果・影響）
  - リスクがある場合はその内容
- 説明後、「実行してよいですか？」と確認する
```

### 各セクションの解説

#### 0. 運用ルール — AIの基本姿勢
- 「すべて日本語で」「専門用語は使わない」「分かりやすく説明する」を明記
- 書かないと英語やコマンドを交えた説明をしてくる

#### 1〜6. 作業スタイル — どう仕事を進めるか
- **Plan ノード**：複雑な作業は先に計画を立ててから動く
- **自己改善ループ**：AIが自分のミスを `tasks/lessons.md` に記録し、同じ失敗を繰り返さない
- **コア原則の3つ**：「シンプルに」「手を抜くな」「余計なことはするな」で勝手に範囲を広げるのを防ぐ

#### 7〜10. 安全ルール — settings.local.json と連動
| ルール | 内容 |
|---|---|
| 7 | ファイルを書き換える前に「いいですか？」と聞く |
| 8 | ファイルを消す操作は原則やらない |
| 9 | ソフトを入れる前に「何を・なぜ・どこに」を説明する |
| 10 | よくわからないコマンドは実行前に日本語で説明する |

## なぜ2つとも必要か

**片方だけでは不十分な理由：**

| | CLAUDE.mdだけ | settings.local.jsonだけ |
|---|---|---|
| 問題点 | AIが長い作業中にルールを見落とすことがある | 「やっていい・ダメ」しか伝えられない。理由を書けない |
| リスク | ルールが「破られる」可能性がある | 未登録の新コマンドはすり抜ける |

**包丁の例え：**
- `settings.local.json` = 包丁に安全カバーを付ける（物理的に刃に触れない）
- `CLAUDE.md` = 「包丁は危ないから気をつけて」と伝える（応用が効く）

両方やって、初めて安心できる = **二重ロック**。

---

## まとめ

やることは2つだけ：

1. `.claude/settings.local.json` を作って、許可と禁止のリストを書く
2. `.claude/CLAUDE.md` を作って、AIへのルールと理由を文章で書く

**非エンジニアだからこそ、この設定をやっておく価値がある。**  
エンジニアは何かあっても自分で元に戻せるが、非エンジニアはそうはいかない。  
先に守りを固めておけば、安心してAIに任せられるし、もっと大胆に使えるようになる。

> ⚠️ この設定は現時点のもの。公式のセキュリティ強化などで変わる可能性あり。  
> 「設定したから完璧」と思わず、定期的に更新し続けること。
